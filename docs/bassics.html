bassics

The following is a list of the built-in types and common terms you will run into when writing Bass programs.

path

Paths are family of types unique to Bass. They represent abstract locations of files or directories.

Bass distinguishes between file and directorie paths by requiring a trailing slash (/) for directories.

(def file ./some-file)
(def dir ./some-dir/)
result
dir

Directory paths can be extended to form longer paths:

dir/file-in-dir
result
./some-dir/file-in-dir

Extending a file path, however, will result in an error.

file/bam
stderr 
error! call trace (oldest first):
  1. (docs):1	file/bam
cannot extend path ./some-file (bass.FilePath) with bam (bass.FilePath)

Throughout this documentation, workloads will be rendered as space invaders. Workloads are technically just objects - click them to reveal their inner structure.

The arguments to a path combiner provide values to be passed to the workload on stdin.

To specify values on argv instead, use (with-args) - or use the $ operative for syntactic sugar:

(def val "hello")

[(-> (.echo) (with-args "hi" val))

All path combiners set their argument list as Workloads are constructed through a series of builder functions provided by the stdlib. Though there is technically an internal schema, it's best to rely on the builder functions.

The -> operative comes in especially handy for building bespoke workloads without wanting to claw your eyes out.

To run a workload, pass it to (run); it will return a stream from which you can read values using (next).

Putting it all together into one big -> pipeline, you get:

(-> (.cat 42) (in-image "alpine") run next)

runtime

An internal component used for running workloads, configured by the user and associated to a platform.

Currently, only Docker is supported - Kubernetes support may come in the future.

workload path

A path paired with the workload that created (or creates) it.

(path (.run) ./some-file)
result

Workload paths can be constructed and passed around to other workloads as first-class values. The runtime will handle the boring mechanical work of mounting it into the container in the right place.

(def touchi-artist
  (-> ($ .touch ./artist)
      (in-image "alpine")
      (path ./artist)))

(-> ($ .ls $touchi-artist)
    (response-from :exit)
    (in-image "alpine")
    run
    next)

Workload paths represent reproducible artifacts - when passed from one workload into another, any paths created by the child workload will be inherently downstream of all of its inputs. The workload path may be exported to JSON format and used to re-create the artifact from scratch with bass -e < wlp.json.

symbol

A name, typically associated to a value in an environment.

Symbols are not typically used as values, though they can be constructed with the (quote) operative.

(quote foo)

environment

A context for evaluation. Basically a scope - it might even be renamed to it someday.

An environment has bindings and a list of parent environments which are queried depth-first when looking for bindings.

null

Everyone's old friend. Used to represent the absense of value where one might typically be expected. Everyone's old friend.

Note: null is a distinct type from an empty list. The only (null?) value is null, and the only (empty?) value is [].

boolean

true or false, but sometimes null.

Boolean values are pretty straightforward - the only catch is that null also counts as false when given to (if) or (not).

Otherwise, all values - including "empty" ones - are truthy.

[(if true :truthy :falsy)
 (if false :truthy :falsy)
 (if null :truthy :falsy)
 (if [] :truthy :falsy)
 (if "" :truthy :falsy)
 (if _ :truthy :falsy)]

lists

Lists are the bread and butter of any Lisp, Bass included.

Bass has two "flavors" of lists which ultimately boil down to the same data structure.

empty list

An empty list is represented by () or []. In a "proper" list, an empty list is always in the final position of a chain of pairs.

pair

Parens, i.e. (a b c), denote a pair - most commonly used as a linked list.

A pair combines a value with another value. When a pair is evaluated, it applies its first value to the second value. With lists, the second value is typically another list, until one finally ends with the empty list.

cons

Square brackets, [a b c], denote a cons - a constructor for pairs.

A cons is a list that evaluates to a pair. It is essentially syntax sugar for constructing list values instead of evaluating combiners.

[1 2 3]
result
(
  1. 1
  2. 2
  3. 3
)
non-list pairs

The second value of a pair isn't always another list - any value may be supplied using the & delimiter. This notation is can be used to call a combiner with a list of arguments:

(def values (list 1 2 3))
(+ & values)
result
6

It also works for cons:

[1 & 2]
result
(1 & 2)
ignore

_ - that's right, just an underscore - is a special constant value used to discard a value when binding values in an environment.

(def (a & _) [1 2 3])

a ; the only binding in the current environment
result
1

_ is also used when null is just not enough to express how absent a value is - for example, to record overarching commentary within a module.

; Hey ma, I'm a technical writer!
_
result
_

A mapping from keywords to values.

{:a 1 :b 2 :c 3}

Objects encode to JSON by converting hyphens to underscores in their keys. This may be slightly surprising but overall it helps with interoperability while keeping Bass code looking Lispy. 

(dump {:foo-bar "hello!"})

}

}

pipes

Pipes encompass sources and sinks.

source

A source of values which may be read with (next).

All scripts can read values from *stdin*, which reads JSON encoded values from stdin.

A source may be constructed from a list of values by calling (stream).

(def vals (stream 1 2 3))
[(next vals)
 (next vals)
 (next vals)]
result
(
  1. 1
  2. 2
  3. 3
)

A source is also returned by (run) to pass along values emitted by the workload.

When (next) hits the end of the stream, an error will be raised. A default value may be supplied as the second argument to prevent erroring.

sink

A destination for values which may be sent with (emit).

All scripts can emit values to the *stdout* sink, which encodes values as JSON to stdout.

(emit "hello!" *stdout*)
(emit 42 *stdout*)
stdout
"hello!"
42
result
null
keyword

A name value, similar to a symbol, only it evaluates to itself instead of fetching a binding from an environment.

Keywords are sometimes used as simple values indicating certain conditions, such as :end to indicate the end of a stream.

(def vals (stream 1 2))

[(next vals :end)
 (next vals :end)
 (next vals :end)]
result
(
  1. 1
  2. 2
  3. :end
)

Keywords are also applicative - they can fetch a value from an object, or fetch a binding from an environment:

(def foo 123)

[(:b {:a 1 :b 2 :c 3})
 (:foo (get-current-env))
]
result
(
  1. 2
  2. 123
)

Keywords may also be chained to the end of a symbol to traverse nested structures:

(def foo {:a {:b 42}})
foo:a:b
result
42
combiner

A value which can be applied to another value (typically a list of arguments) to perform some computation and return another value.

operative

A combiner which receives the applied arguments, unevaluated, along with the dynamic environment of the caller.

Operatives are defined with the (defop) operative or constructed with (op).

(defop quote-with-env args env
  [args env])

(quote-with-env a b c)
result
(
  1. (
    1. a
    2. b
    3. c
    )
  2. <env>
)
applicative

A combiner which wraps an underlying operative and evaluates its arguments before passing them along to it.

Applicatives, typically called functions, are defined with the (defn) operative or constructed with (fn).

(defn inc [x]
  (+ x 1))

(inc 41)
result
42
